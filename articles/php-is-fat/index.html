PHP is fat and really, really stupid

PHP is one of the most verbose mainstream language for expressing typed, structured code. Yet it earned no such reputation because the community has convinced itself the verbosity is a virtue!

Explicit is better than implicit. That principle applies when implicit behaviour is surprising. It does not apply when the implicit behaviour is the only possible behaviour, enforced by the spec.

This is just noise with a philosophical label stuck on after the fact.

Explicit Content!

Take the PHP documentation's own example for constructor promotion. Here it is in modern idiomatic PHP: final readonly, promoted properties, two methods, each body a single expression: 

final readonly class Point
{
    public function __construct(
        public float $x,
        public float $y,
    ) {}

    public function translate(float $dx, float $dy): self
    {
        return new self($this->x + $dx, $this->y + $dy);
    }

    public function distanceTo(Point $other): float
    {
        return sqrt(($other->x - $this->x) ** 2 + ($other->y - $this->y) ** 2);
    }
}

Wow! Look at the width of those methods! And this is not even a worst-case example with extra modifiers or dare I say when generics finally arrive... NEVER!

Side note: a Point should not be a class at all. Classes carry object identity baggage and live on the heap, huge overhead!! A coordinate pair is a value with no identity, and creating one should cost nothing, but PHP has no structured value types, so here we are.

Should we use Go instead?

Wait don't Go! You should only Go... here :-) . It's not Go-Over yet. PHP is not a legacy language!!

This is easily fixed in PHP-src! (Oh yes I'm sure everyone will agree and not be mad!) Just yeet the redundant keywords. The same rules still apply, but now the code is much easier to read. 

final readonly class Point
{
    __construct(
        public float $x,
        public float $y,
    ) {}

    translate(float $dx, float $dy): self
    {
        return new self($this->x + $dx, $this->y + $dy);
    }

    distanceTo(Point $other): float
    {
        return sqrt(($other->x - $this->x) ** 2 + ($other->y - $this->y) ** 2);
    }
}

Inside a class every method is implicitly public and function is identifiable from context. Neither keyword adds information.

Properties could Go further with a rule borrowed from many other langs: a bare typed property is private by default. public is only written when you mean to expose something, which is when it actually carries information: 

final readonly class Range
{
    public float $min, $max;

    contains(float $value): bool
    {
        return $value >= $this->min && $value <= $this->max;
    }

    clamp(float $value): float
    {
        return max($this->min, min($this->max, $value));
    }
}

A class with mixed visibility needs no special syntax. Bare properties are private, public marks the API surface: 

class Connection
{
    public string $host;
    public int $port;

    string $username;
    string $password;
}

The comma syntax could also Go further. Once a type is established on a line, subsequent names in the list could drop the sigil and type, and mixing types in a single declaration should also be possible. Blocks give you a way to group a visibility without repeating it: 

class Example
{
    // same type, shorthand names after the first
    public float $x, y, z;

    // mixed types in one line
    public float $width, int $count, string $label;

    // visibility block, public stated once, covers everything inside
    public {
        float $min, max;
        string $name;
    }

    // bare properties are private, no keyword needed
    string $secret;
    DateTime $createdAt;
}

It is only getting worse!!1

PHP 8.4 added property hooks, they are pretty cool. But the verbosity compounds!!! The public modifier now sits at the front of a block that already has a default value, a get body, and a set body. It is the least interesting word on the page: 

class Example
{
    bool $modified = false;

    public string $foo = 'default'
    {
        get => $this->modified ? $this->foo . ' (modified)' : $this->foo;
        set => $this->applyFoo($value);
    }

    private function applyFoo(string $value): void
    {
        $this->foo = strtolower($value);
        $this->modified = true;
    }
}

Oh and now properties can be put on interfaces, cool! Makes sense given it's just a syntactic sugar for a getter and an optional setter: 

interface HasId
{
    public int $id { get; }
    public string $name { get; }
}

But wait, you have to write public?? There is no world in which those properties are not public, it's an interface for God's sake! The modifier communicates nothing.
